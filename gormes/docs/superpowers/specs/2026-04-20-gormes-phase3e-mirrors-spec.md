## 1. Purpose

Phase 3.E completes the Black Box memory system by closing three orthogonal gaps:

1. **Operational Visibility** — Human-readable mirrors of machine-persistent state (session index, tool audit, insights, transcripts)
2. **Memory Decay** — Biological forgetting: weights attenuate, old facts fade from recall without deletion
3. **Cross-Chat Synthesis** — One operator, multiple platforms, unified graph; "what is Juan working on?" returns facts from Telegram, Discord, and Slack

These 8 deliverables are separable specs that can ship independently. Priority order is ranked by operational pain vs. implementation effort.

### 🟢 Lower Priority / Future

#### 3.E.6 — Memory Decay
**Problem**: Graph grows forever; stale facts compete with fresh facts in recall.  
**Solution**: Weight attenuation on relationships + `last_seen` tracking. Old facts fade without deletion (reversible, audit-preserving).  
**Implementation**: ~200 lines.

**Algorithm**:
```go
// Decay applies exponential weight reduction based on time since last_seen
func (r *Relationship) Decay(now time.Time, halfLife time.Duration) {
    age := now.Sub(r.LastSeen)
    periods := float64(age) / float64(halfLife)
    r.Weight = r.InitialWeight * math.Pow(0.5, periods)
}
```

**Configuration**:
```yaml
memory:
  decay:
    enabled: true
    half_life_days: 30        # Weight halves every 30 days of non-use
    minimum_weight: 0.1       # Floor to prevent complete disappearance
    reactivate_boost: 1.5       # Multiplier when entity is seen again
```

**TDD Test Cases**:
1. Test `Decay()` reduces weight correctly for given age
2. Test floor prevents weight going below minimum
3. Test `Reactivate()` boosts weight when entity reappears
4. Test decay doesn't affect source-of-truth (SQLite stores original)
5. Test query-time decay (recall only, no write-back)

#### 3.E.7 — Cross-Chat Synthesis
**Problem**: One operator has Telegram, Discord, Slack sessions — but graph is fragmented by `chat_id`.  
**Solution**: Graph unification across `chat_id` boundaries. Query "what is Juan working on?" returns facts from all platforms.  
**Implementation**: ~300 lines. Requires `user_id` concept above `chat_id`.

**Architecture**:
```
┌─────────────────────────────────────────┐
│           User (operator_id)            │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐  │
│  │Telegram │ │ Discord │ │  Slack  │  │
│  │ chat_1  │ │ chat_2  │ │ chat_3  │  │
│  └────┬────┘ └────┬────┘ └────┬────┘  │
│       └───────────┼───────────┘       │
│                   ▼                     │
│         Unified Entity Graph            │
│    (all facts visible to all chats)     │
└─────────────────────────────────────────┘
```

**TDD Test Cases**:
1. Test cross-chat recall returns entities from multiple `chat_id`s
2. Test entity deduplication (same entity mentioned in two chats)
3. Test privacy boundaries (User A cannot see User B's facts)
4. Test `user_id` persistence across session resume
5. Test migration path from single-chat to cross-chat mode

## 4. Implementation Principles

All 8 deliverables follow these principles:

### 4.1 Fire-and-Forget Logging
- All mirrors use background goroutines
- Main loop latency impact: zero
- Channel-based async writes with bounded buffers
- Graceful degradation: if disk full, log error and continue

### 4.2 Atomic Writes
- All file mirrors use temp-file + rename pattern
- Readers never see partial writes
- No file locking required

### 4.3 Source of Truth Remains SQLite
- Mirrors are read-only exports
- No bidirectional sync complexity
- Regeneratable from source (delete YAML, it recreates)

### 4.4 XDG Directory Compliance
```
~/.local/share/gormes/
├── sessions/
│   ├── sessions.db          (bbolt — source of truth)
│   └── index.yaml           (3.E.1 mirror)
├── memory/
│   ├── memory.db            (SQLite — source of truth)
│   └── USER.md              (3.D.5 mirror — shipped)
├── tools/
│   └── audit.jsonl          (3.E.2 mirror)
├── insights/
│   └── usage.jsonl          (3.E.5 mirror)
└── transcripts/
    └── exported/            (3.E.3 export destination)
```

### 4.5 Configuration-First
```yaml
# ~/.config/gormes/config.yaml
mirrors:
  session_index:
    enabled: true
    interval_seconds: 30
    path: "~/.local/share/gormes/sessions/index.yaml"
  insights:
    enabled: true
    daily_summary: true
  tool_audit:
    enabled: true
    retention_days: 90
    sensitive_args_redacted: true
  decay:
    enabled: false  # opt-in, requires semantic model for relevance scoring
    half_life_days: 30
```

## 7. Acceptance Criteria

### 7.1 Session Index Mirror
```gherkin
Given a running Gormes instance with 3 active sessions
When the operator runs `cat ~/.local/share/gormes/sessions/index.yaml`
Then they see all 3 sessions with metadata (platform, chat_id, message_count)
And the file updates automatically every 30 seconds
```

### 7.2 Insights Audit Log
```gherkin
Given a session that consumed 1000 tokens and called 5 tools
When the session closes
Then a JSON line is appended to `~/.local/share/gormes/insights/usage.jsonl`
And the line contains token counts, tool counts, and estimated cost
```

### 7.3 Tool Audit Log
```gherkin
Given a tool execution that took 250ms
When the tool completes
Then an audit record is written asynchronously
And sensitive arguments are redacted per config
```

### 7.4 Transcript Export
```gherkin
Given a session with 50 message turns
When the operator runs `gormes session export <id> --format=markdown`
Then a Markdown file is created with all turns formatted
And the output is human-readable without SQLite tools
```

### 7.5 Memory Decay
```gherkin
Given a relationship with weight 1.0 last seen 60 days ago
When decay is enabled with 30-day half-life
Then the effective weight in recall queries is 0.25
And the stored weight in SQLite remains 1.0
```

### 7.6 Cross-Chat Synthesis
```gherkin
Given an operator with Telegram chat A and Discord chat B
When they mention "Kubernetes cluster" in chat A
And later query "what did I say about Kubernetes?" in chat B
Then the system returns the entity from chat A
```

## 8. Future Considerations (Not in 3.E)

These items are intentionally deferred to later phases:

- **Real-time Sync**: Mirrors are pull (read), not push (sync). WebSocket live updates out of scope.
- **Remote Storage**: S3/GCS backup of mirrors — use `rclone` or similar outside Gormes.
- **Compression**: JSONL files can be large; gzip rotation is future work.
- **Query Interface**: SQL interface over JSONL — use `jq` or `sqlite3` import.
- **Visual Dashboards**: Grafana/Prometheus export — parse JSONL externally.

## 9. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Disk full from audit logs | Medium | High | Configurable retention + rotation |
| YAML corruption on crash | Low | Medium | Atomic writes (temp+rename) |
| Privacy leak in tool audit | Medium | High | Default redaction + audit review |
| Decay removes critical facts | Low | High | Minimum weight floor + reversibility |
| Cross-chat privacy boundary failure | Low | Critical | Strict user_id isolation tests |
| Binary bloat | Low | Medium | Feature flags; compile out unused mirrors |
